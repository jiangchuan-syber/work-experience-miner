# UI 设计（用户视角）

## 核心立场

设计服务于**用户此刻要完成的任务**，不是把系统能力全部摊开。每个控件都要能回答：**谁、在什么场景、要完成什么、多久能完成**。

优先级（冲突时按此取舍）：

1. **任务可完成** — 主路径无断点
2. **主操作始终可达** — 高频输入/确认不随滚动消失（见「交互」）
3. **信息可扫读** — 3 秒内找到关键答案
4. **结构扁平** — 少层级、少模态、少记忆负担
5. **风格一致** — 像同一款产品
6. **区域不打架** — 固定栏、浮层、侧栏互不挡内容

---

## 交互（必审）

交互失败常表现为「功能在，但用起来费劲」。改版或评审时**单独过一遍本段**，不可只改视觉。

### 原则

| 原则 | 含义 | 失败信号 |
|------|------|----------|
| **主 CTA 不随滚走** | 聊天输入、提交、保存等应贴在**视口底**或独立滚动区外 | 用户读长文后要滚到底才能输入 |
| **消息与输入分离** | 历史/正文可滚；输入条单独一层 | 输入框夹在消息中间 |
| **控件要说人话** | 按钮/入口用完整词（「管理经历」「使用说明」） | 单独的 `?`、`⋯`、无标签图标 |
| **同一意图单一入口** | 选一个对象、切一段经历等只保留一处控件 | 「对话选用」+「设为对话」并存 |
| **状态一处说清** | 当前对象、是否就绪用标题下一行展示 | 同义状态在页头、卡片、底栏重复三遍 |
| **调试与学生路径隔离** | trace 恢复、重建索引等进「高级 · 调试」折叠 | 侧栏默认露出开发路径 |

### 滚动与固定底栏（对话型页面）

```
┌─ 可随页滚动的内容 ─────────────────┐
│  经历编辑、对话气泡、列表…          │
│  （末尾留 padding-bottom ≥ 输入条高）│
└────────────────────────────────────┘
┌─ 固定在视口底（不随上文滚走）──────┐
│  st.chat_input / 主操作条            │
└────────────────────────────────────┘
```

- **禁止**把输入条放在长文档流中间，仅靠 `position: sticky` 且父级整页滚动——长内容时仍会滚走。
- **推荐**：DOM 上输入条在页面**最末** + `position: fixed` 对齐主内容区宽度；侧栏收放时需重算 `left/width`（本项目见 `app.py` 中 `_pin_chat_input_bar`）。
- **禁止**用更高 `z-index` 让经历卡片等内容**盖住**底栏输入；内容区 z-index 应低于固定输入条。
- 消息容器预留 `padding-bottom`，避免最后一条气泡被输入条挡住。

### 首屏与顶栏

- 压缩 Streamlit 默认 `block-container` 上内边距与顶栏占位；勿在标题旁单独占一列放「?」。
- **禁止** `display: none` 隐藏整个 `[data-testid="stToolbar"]`，否则侧栏收起后无法展开；仅需弱化 Deploy 时用更精确选择器。
- **禁止**再做一个「打开侧栏」类按钮：Streamlit 已有左上角 **⟩**（`stExpandSidebarButton`），重复入口违背「同一意图单一入口」。
- 技术说明用 **`st.expander("使用说明")`**，默认折叠；状态用 **`st.caption` 一行人话**（如「对话已就绪 · 当前对话：第 N 段 · 公司名」）。

### 交互自检（与遮挡检查一起做）

```
- [ ] 不滚动页面即可看到并聚焦主输入/主按钮
- [ ] 滚到任意长内容中间，主输入仍贴在视口底
- [ ] 所有可点击入口均有可读中文标签（无孤立 ? / ⋯）
- [ ] 同一操作不存在两个等价入口
- [ ] 当前选中对象在页头一处可读，底栏不重复长状态
- [ ] 侧栏无默认展开的调试项
```

## 工作流

接到 UI 相关需求时，按顺序执行（可跳过不适用的步骤，但不可跳过「字段审计」「交互」与「遮挡检查」）：

```
1. 定义用户任务 → 2. 信息架构 → 3. 布局与层级 → 4. 交互与主 CTA → 5. 字段与文案 → 6. 一致性 → 7. 遮挡与响应式 → 8. 交付说明
```

### 1. 定义用户任务

用一句话写清：**用户打开此页/此区域要完成什么**。列出主路径（≤3 步）与次要路径；其余能力降级或收起。

### 2. 信息架构

- 按**用户心智**分组，不按数据库表或 API 结构分组
- 同一屏只服务**一个主任务**；次任务用 Tab、折叠、二级页，勿堆一屏
- 建立**视觉层次**：主信息 > 辅助说明 > 元数据/调试信息

### 3. 布局与扁平化

**扁平**指认知与操作路径短，不是「把所有东西铺在首屏」。

| 原则 | 做法 | 避免 |
|------|------|------|
| 少层级 | 主内容默认可见；高级项进「更多」/侧栏 | 默认三层以上导航才到目标 |
| 少模态 | 能内联编辑就不弹窗；能侧栏就不全屏遮罩 | 连环弹窗、无 ESC/返回 |
| 少跳转 | 相关操作同屏或同栏完成 | 填一个字段跳一页 |
| 状态可见 | 当前步骤、保存状态、错误贴近字段 | 仅 toast，用户不知改哪 |

### 4. 交互与主 CTA（必做）

对照上文「交互」章节：主输入是否固定视口底、消息/输入是否拆分、入口是否有中文标签、是否存在双入口。改布局优先于只加 CSS。

### 5. 字段审计（必做）

对**每一个**字段/控件问：

| 问题 | 否 → 处理 |
|------|-----------|
| 用户**此刻**需要它吗？ | 隐藏、折叠、或移到设置/详情 |
| 用户能**一眼**看懂标签与单位吗？ | 改文案、加示例、去 jargon |
| 与已有信息重复吗？ | 合并或只读展示一处 |
| 空值/错误时用户知道怎么办吗？ | 就地提示 + 可操作建议 |
| 是「填一次长期有效」还是「每次会话」？ | 默认值、记忆、少打扰 |

**默认展示**：主任务必填 + 高频可选。**默认隐藏**：低频配置、开发者选项、导出格式细节。

### 6. 信息获取便捷性

- **F 型扫读**：关键结论左上或标题下第一屏；列表用对齐列，勿靠颜色 alone 区分
- **一屏一答**：用户常问的问题（状态、进度、下一步）不必滚动找
- **就近原则**：标签紧贴控件；错误在字段下；帮助 ≤1 句或 tooltip，长文链到文档
- **可比性**：同类项相同控件类型与顺序（例如日期一律 date picker）

### 7. 风格统一

在同一产品/仓库内对齐（无设计系统时先定最小集）：

- **间距**：8px 网格（或 4/8/12/16/24）
- **字号层级**：标题 / 正文 / 辅助 / 代码 各一档，勿临时加大
- **颜色语义**：主色、成功、警告、错误、禁用 固定用途
- **组件**：主按钮每区最多一个；次要操作用 text/link；危险操作用确认
- **文案**：同一概念一个词（如「导出」vs「下载」二选一）
- **空态/加载/错误**：三种状态有统一模板，勿裸 `st.write` 或空白

改版时先 grep/搜索现有页面，**复用已有 class、组件、侧栏模式**，再考虑新样式。

### 8. 功能互不遮挡（必做）

检查清单：

- [ ] **固定底栏**是否始终可操作，且未挡主内容最后一屏？消息区有 `padding-bottom`
- [ ] **内容区 z-index** 是否低于固定输入条（勿让卡片叠在输入条上）
- [ ] **固定顶栏/侧栏**是否与主区滚动独立？展开侧栏时主区是否仍可读
- [ ] **浮层**是否被父级 `overflow:hidden` 裁切
- [ ] **移动端/窄屏**：侧栏与底栏是否叠在同一角；可点击区域 ≥44px
- [ ] **Streamlit**：改 `app.py` 布局后是否在浏览器中**滚长对话**实测输入条是否跟滚

发现遮挡：**改布局优先于加 z-index**；确需浮层时保证可关闭、不挡主 CTA。

## 本仓库布局约定（工作经历挖掘 / Streamlit）

实现以 `app.py` 为准；改 UI 时保持下列交互契约：

| 区域 | 职责 | 交互要点 |
|------|------|----------|
| **侧栏** | API Key、**对话历史**、（底部）`高级 · 调试` | 会话列表**不**单独 `max-height` 滚动，随侧栏整体滚 |
| **页头右上** | **简历导入**（Popover）· **导出** | 与标题同一行；导入在导出左侧 |
| **主区（全宽）** | 对话选用、经历卡、对话消息 | 当前段 expander 默认展开；**无**卡片内「设为对话」 |
| **页头** | 标题 + caption + `使用说明` expander | 无 `?` 列；技术细节不进 caption |
| **底栏** | `jx_chat_input_bar` + `_pin_chat_input_bar` | 与 `jx_chat_scroll` 分离；fixed 贴视口底 |

新增字段或按钮前：grep `app.py` 是否已有等价入口；避免第三处「选经历」或第二处状态文案。

## 实现前自检（30 秒）

```
- [ ] 能说出本页/本区的「用户一句话任务」
- [ ] 首屏可见内容支持主路径，无「为了完整而完整」的字段
- [ ] 主输入/主 CTA 不随长文滚动离开视口（已浏览器滚测）
- [ ] 无孤立 ? / ⋯；无同一意图的双入口
- [ ] 无两个固定层同时占同一视觉带（底+底、顶+顶）
- [ ] 术语、按钮层级、间距与相邻页面一致
```

## 交付时的说明格式

完成 UI 改动后，用简短段落回复（不必冗长）：

```markdown
## 用户任务
[一句话]

## 信息结构
[主区 / 侧栏 / 底栏各承担什么]

## 字段决策
| 字段/区域 | 保留/隐藏/合并 | 理由 |
|-----------|----------------|------|

## 交互
[主 CTA 固定方式、消息/输入拆分、入口命名]

## 扁平化与遮挡
[层级、固定栏处理、窄屏注意点]

## 一致性
[复用的组件/样式/文案约定]
```

评审他人 UI 时，用同一模板输出**问题 + 建议**，按严重度标注：

- **阻断**：无法完成任务或关键信息被挡
- **困扰**：多步、难找、易误操作
- **打磨**：一致性、文案、间距

## 反模式（见到就改）

| 反模式 | 为什么差 | 改法 |
|--------|----------|------|
| 字段大全屏 | 制造焦虑、拖慢主路径 | 分步、折叠、按场景显示 |
| 侧栏塞满配置 | 主区变摆设 | 侧栏只放会话级/导入导出；其余进设置 |
| 双主按钮 | 用户不知点哪个 | 一主一辅 |
| 错误只在顶部 banner | 用户不知哪项错了 | 字段下就地错误 |
| 固定底栏无留白 | 最后几行永远点不到 | 主容器 bottom padding |
| 每页一套 CSS | 风格分裂 | 抽 tokens / 共用组件 |
| 输入条在文档流中 | 长对话后滚到底才能输入 | 消息/输入拆分 + fixed 底栏 |
| 无标签 `?` / `⋯` | 用户不知用途 | 使用说明 expander、管理经历等有词按钮 |
| 双入口选同一对象 | 认知分裂 | 只保留置顶「对话选用」等一处 |
| 内容 z-index 高于底栏 | 输入被卡片盖住 | 内容区 z-index 低于输入条 |

## 技术栈提示（简）

- **Streamlit**：侧栏轻量；对话页拆 `jx_chat_scroll` / `jx_chat_input_bar`；底栏 fixed 需对齐 `stMain` 宽度（见 [patterns.md](patterns.md)）；改版后**浏览器滚长对话**验收
- **Web SPA**：路由深度 ≤2 到主任务；全局壳（nav + content）与页内状态分离
- **表单**：必填标 * 一次；提交失败 focus 到第一个错误字段

详细模式库见 [patterns.md](patterns.md)（可选阅读）。
