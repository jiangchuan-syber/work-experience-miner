# 经历表达：模板与改稿规范

约定：先有事实与故事（参见 [经历挖掘-对话引导.md](./经历挖掘-对话引导.md)），再用本文把材料压成 **简历要点**、**面试 STAR**；与岗位差异化指标对齐时使用 `01`～`07`。  
本文同时作为 **协助改稿的输出格式**：模型或导师按最后一节 **[改稿交付结构]** 给出修改结果，避免口径散乱。

---

## 一、经历模块在简历里的骨架

每一块「实习 / 全职 / 重点项目」建议统一为三层信息（复制粘贴时保持同级缩进层级一致）：

```text
{组织名称} | {地点（可选）}    {起始年月}～{结束年月}
{职位 / 课题角色}｜{业务线或部门（可选，脱敏则用行业词替代）}

• {要点 1}
• {要点 2}
• {要点 3}
```

**脱敏**：产品名可用「某政企客户 / 内部效率平台」等；禁止写密钥、完整内网拓扑、涉密项目名称。

---

## 二、简历单行要点（Bullet）写法规范

### 2.1 推荐句式

由内到外依次为：

```text
[可选：角色前缀] + 行为动词 + 对象/范围 + 手段或约束 + 可验证结果
```

**角色前缀**（仅当易被误解分工时使用）：独立完成 / 作为主开发 / 在 mentor 指导下 等。

**行为动词**：过去时语感（「负责」「搭建」「优化」「将…」「推动」「沉淀」）；避免「参与」「协助」单独打头且无后续落点。

### 2.2 硬性规则

| 规则 | 说明 |
|------|------|
| **一事一条** | 一条 bullet 只承载一个主轴成就或一类职责；强拆成两条更清晰。 |
| **少形容词** | 不写「认真负责」等泛泛品质；用事实替代。 |
| **结果可核对** | 优先数字（绝对值、比率、Δ、频次、延时）；次之写清**前后对比或替代指标**（如「由人工群发改为工单触发」）。 |
| **数字诚实** | 无精确统计时写明「约」「抽样」或使用区间；不得虚构业务结果。面试需能解释口径。 |
| **长度** | 中文单行以约 **35～85 字** 为宜（含标点）；超长则拆分。 |

### 2.3 量化优先顺序（可写不进再降级）

1. 业务：收入、转化率、成本、工时、工单量、投诉率、 uptime 等。  
2. 工程：延时、吞吐、Bug 逃逸率、覆盖率、构建时间、发布频次。  
3. 协作：文档被引用次数、培训场次、采纳的 RFC 数量（需在面试说清）。

### 2.4 慎用与禁用

| 慎用 | 原因 |
|------|------|
| 「熟悉 / 了解」堆砌 | 与经历叙述重复且无过程。 |
| 「参与项目」戛然而止 | 需补「你做了什么决策/交付物」。 |

| 禁用 | 原因 |
|------|------|
| 泄密与客户可识别信息 | 合规与职业道德。 |
| 无法解释的夸张 | 如一概「提升 80%」而无分母与时间窗。 |

### 2.5 Bullet 成稿自检（模型改稿文末应内化核对）

- [ ] 删掉「的团队很优秀」等与个人贡献无关的信息。  
- [ ] 「若没有我」是否仍可能发生？若会发生，本条是否降格为次要或改写角度。  
- [ ] 是否至少具备一类证据：工单、PR/MR、版本号、周报、评测表、会议纪要 action。  
- [ ] 动词是否与目标岗位 JD 的关键词可对齐。

## 三、STAR 口述（面试）写法规范

用于行为面试：**同一故事**可同时拆成多条 bullet（突出不同维度）**，**STAR 保持一个完整叙事**。

### 3.1 四段字数与要点

| 段落 | 篇幅（中文合计参考） | 要点 |
|------|----------------------|------|
| **S** Situation | 2～4 句 | 背景 **+ 你在其中的位置**；忌长篇公司史的。 |
| **T** Task | 1～2 句 | 目标 **与约束**（时间、质量、干系人、冻结范围）。 |
| **A** Action | **主体** | 拆成 **3～6 句**，每句主语尽量是「我」，写**决策与操作**而非团队流水账；技术岗带方法与工具（如链路追踪、回放数据、对照实验）。 |
| **R** Result | 1～3 句 | **结果数值或定性 + 复盘一句**（可写「不足与改进」显现成熟度）。 |

**整段总长**：面试稿约 **280～520 字**；口语可再在数字上停顿展开。

### 3.2 STAR 禁忌

- 把 **Situation** 写成行业科普。  
- **Action** 里「我们」虚化到看不出个人动作——需强行落到「我做了 X，因此 Y」。  
- **Result** 仅「锻炼了能力」而没有对业务/工程的可见影响（至少说明可验证的中间态）。

## 四、与其它材料的对应关系（可选小节）

投稿内推或英文简历时：

- **中文 bullet** ↔ **English one-liner**：保持事实一致；动词可参考岗位 JD（impact-driven）。  
- **作品集 / Git**：在 bullet 末可加「（见 README / Demo）」，避免在正文贴长链接。

---

## 五、与岗位知识库对齐

改稿时在内部（不必全部写进简历）维护一张 **对齐表**：

| 知识库文件 | 命中的小节标题（复制 `###` 行） | 用户事实一句（不脱敏草稿可另存） | 证明材料 |
|------------|----------------------------------|-------------------------------------|----------|
| 例如 `01-软件开发与交付.md` | 后端开发工程师 / 实习 | … | MR #123 |

**缺口**：表中若某小节无事实，不得在简历中编造对应话术；标注为「待补充」并从用户追问。

## 六、协助改稿：模型 / 导师 **必须遵守的交付结构**

用户粘贴一段**原始口述或草稿**并说明 **目标场景**（仅简历 / 仅面试 / 两者都要）与 **意向岗位文件名**（如 `01`、`07`）时，回复请 **严格按下述标题与顺序** 输出（可删「不需要」的子节须在首行注明目标场景）。

```text
## 原文摘要
（一句话复述用户主旨，不写评价。）

## 假设与待定事实
（列出改稿中用到的但未在用户原文中出现的数字或结论；一律标为「待用户确认」，不得伪装成已定事实。若无则写「无」。）

## 简历要点｜修订版
（输出 1～5 条「• …」，每条遵循本文第二节硬性规则；与目标岗位动词对齐。）

## STAR｜口述稿精简版
（可选：仅当用户要面试或直接要求时输出；遵循第三节。）

## 对齐岗位指标
- 文件名与章节：……
- 可支撑的特殊指标条目：……（摘标题或概括为短句）
- 当前证据强度：强 / 中 / 弱（并各用一句话说明理由）

## 待补充清单
（用户需补的具体信息：量化口径、时间段、上下游、权限范围等）

## 润色说明｜极简
（不超过 3 条：改了什么句式、为何不采用某写法，不写冗长夸夸。）
```

**模型纪律**：

1. **不捏造** internship 成果、评比名次、口径未给出的百分比。  
2. 推断组织分工时加注「（推断，请确认）」。  
3. 若文本涉密疑似，改为行业级描述并提示用户自行替换。

## 七、与对话引导的流程衔接

1. [经历挖掘-对话引导.md](./经历挖掘-对话引导.md) 四条锚打完 → 草稿事实。  
2. 本文第二节、第三节 → 成稿。**改稿一律走第六节交付结构**。  
3. `01` / `02` / `03` / `07`（及后续扩充文件）→ 对齐与查漏补缺。
