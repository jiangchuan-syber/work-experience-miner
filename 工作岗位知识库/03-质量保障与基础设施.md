# 质量保障与基础设施 · 岗位特殊评价指标

> 共通维度见 [00-通用底座与条目模板.md](./00-通用底座与条目模板.md)。

---

### 测试工程师 / 质量保证（QA）

- **典型产出**：需求/用户故事 → **可追溯**测试用例矩阵；缺陷单（含附件与环境）；回归范围说明与**风险声明**；上线 checklist 与签字记录。
- **特殊评价指标**
  - **用例设计（及格线）**
    - 是否用等价类、边界、状态转换、异常/权限路径**系统化**覆盖，而非随机点测。
    - 每条用例能否链回**需求 ID/验收标准**；是否标明**数据前提**与清理步骤。
  - **缺陷质量**
    - 复现：步骤编号、期望/实际、**日志 snippet、截图、版本号、账号角色**是否一次给齐。
    - 分级：是否按业务影响与用户规模区分严重级；是否避免「全都是 urgent」。
  - **风险与排期**
    - 时间不够时是否明确**冒烟集合**（核心动线）与**已知未测范围**并排优先级。
    - 是否与研发对齐**修复验证窗口**，避免「修了但未验就上线」。
  - **协作**
    - 是否推动「一次性修完」：附根因假设、是否偶现、是否需补充监控。
- **常见证据**：用例覆盖率对照表、缺陷 reopen 率、遗漏缺陷复盘 action、上线后 24h 观察记录。

### 测试开发（SDET）工程师 / 实习

- **典型产出**：稳定 CI 中的自动化套件；Page Object/Fixture 库；flaky **分类与隔离**文档；失败截图与 trace。
- **特殊评价指标**
  - **稳定性（及格线）**
    - 是否区分环境 flaky 与脚本竞态；是否用语义等待/轮询替代硬 sleep（除极少数场景且注释原因）。
    - 重试策略是否**克制**（不掩盖真实 bug）；是否对间歇失败开 issue。
  - **投资回报**
    - 自动化是否优先落在**高频回归、高风险分支、曾重复手工测 N 次的区域**。
    - 是否能量化：节省人天/缩短反馈时间（哪怕粗略）。
  - **断言与契约**
    - 断言是否锚定**业务不变量**（金额、状态枚举），而非易碎 CSS 选择器（除非无更好 API）。
    - 是否与研发约定**测试数据**生命周期（创建/清理）。
  - **可扩展性**
    - 新人能否在文档指引下 **加一条用例** 而不复制粘贴整文件；公共步骤是否抽取。
- **常见证据**：CI 绿率趋势、平均修复 flaky 时间、流水线artifact、重构 PR 说明。

### 运维 / DevOps / SRE / 实习

- **典型产出**：部署/回滚脚本或 pipeline 片段；IaC 变更 MR；告警规则与 **runbook**；变更单；事故时间线与复盘（参与部分）。
- **特殊评价指标**
  - **变更纪律（及格线）**
    - 是否在窗口内变更；是否双人复核或按团队要求走 approval；是否有**回滚演练**或一键回滚验证记录。
    - 变更单是否含：影响范围、观测指标、rollback 条件。
  - **可观测性**
    - 报警：是否 **symptom-based**（用户可见）且附 runbook 链接；是否避免同一故障多路重复轰炸。
    - 仪表盘：是否区分「业务黄金指标」与「基础设施噪声」。
  - **容量与韧性**
    - 高峰前是否检查 HPA/限流/熔断/队列堆积预案；是否知道**降级开关**在哪里开。
  - **事故参与**
    - 是否能按 playbook 拉 trace/log/metric；是否记录时间线决策；是否事后补**行动项 owner**。
- **常见证据**：CHANGE 记录、On-call 工单、MTTR/MTTD 报表（若有）、演练签到表。

### DBA（数据库管理员）/ 数据库方向实习

- **典型产出**：慢 SQL 分析与**改写建议**或索引变更 MR；备份/恢复检查记录；权限与工单（grant/revoke）；DDL 影响评估（行锁、时长估算）。
- **特殊评价指标**
  - **变更安全（及格线）**
    - DDL 是否在低峰；是否评估**锁等待**与执行计划；大表是否走 online schema 或分批。
    - 是否在业务可接受窗口沟通**只读/主从延迟**风险。
  - **备份与恢复**
    - 是否知道 RPO/RTO；是否有**最近一次**恢复演练或「可恢复性」书面说明。
  - **权限与合规**
    - 生产访问是否**工单化**；导出是否脱敏与最小权限；个人账号与运维账号是否分离。
  - **性能方法论**
    - 是否系统化：explain、锁、缓冲池、归档/分区策略；问题是否闭环到**根因**而非反复加索引。
- **常见证据**：变更审批、慢查询 TopN 闭环工单、备份校验截图、权限审计抽样。

### 网络安全工程师（安全研发 / 攻防 / 合规方向）/ 实习

- **典型产出**：漏洞报告（含 CVSS/影响面/复现条件）；WAF/规则 PR；风险评估表；应急步骤草稿（在授权内）；合规差距清单（不含敏感细节外泄）。
- **特殊评价指标**
  - **授权与边界（一票否决项）**
    - 是否在**书面授权**范围与时间内测试；高危命令是否有复核；是否禁止对生产未授权扫描。
  - **报告质量**
    - 是否写清：**影响资产、可利用条件、数据可达性、修复优先级建议、临时缓解**。
    - 是否区分已验证与推测，并标注**不确定性**。
  - **工程化**
    - 规则/脚本是否进 CI 或定期任务而非一次性；误报是否有**反馈闭环**。
  - **沟通**
    - 对外/对客户表述是否经法务或模板审阅；是否避免过度承诺「绝对安全」。
- **常见证据**：漏报误报复盘、规则命中率（脱敏）、Ticket 关闭记录、培训/分享材料（可公开版）。
