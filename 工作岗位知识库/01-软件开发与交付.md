# 软件开发与交付类 · 岗位特殊评价指标

> 共通维度见 [00-通用底座与条目模板.md](./00-通用底座与条目模板.md)。

---

### 后端开发工程师 / 实习

- **典型产出**：合并进主干的 MR/PR；OpenAPI/Proto 或等价接口说明；功能说明与**回滚/开关**说明（若涉及）；关键路径的单元/集成测试或测试分工记录。
- **特殊评价指标**
  - **接口契约（及格线）**
    - 入参：必填/类型/边界（分页上限、枚举）是否与文档一致；非法请求是否返回**稳定**错误结构与可 log 的 trace/request id。
    - **幂等**：支付、下单、库存类写操作是否用业务幂等键或去重表；重复提交的语义是否写进 MR 描述。
    - **破坏性变更**：字段删除/语义变更是否标版本、双写期或迁移步骤；是否 `@` 消费方并更新契约文件。
  - **数据与并发（按业务复杂度选用）**
    - 是否写明**事务边界**（哪些步骤同学务、何时提交）；典型竞态（重复领取、超卖）是否至少有一次 mentor 认可的取舍说明。
    - 是否有**慢查询排查痕迹**：explain、索引建议或「已知 N+1、计划下迭代修」的备注；大表扫是否在后台任务或限制范围。
    - 缓存（若有）：key 规则、TTL、与 DB 的最终一致性说明是否可读，而非「先写了再说」。
  - **可运维性**
    - 日志：异常是否带**业务主键**（orderId/userId）与错误码；是否避免打印 PII/完整 token。
    - 配置：环境变量与密钥是否分离；feature flag 默认值与关闭路径是否交代。
    - 发布：是否在预发/staging 跑过**主路径**；灰度范围与观测指标（错误率、延迟）是否写在发布单或 MR。
  - **质量习惯**
    - 核心业务是否覆盖单测/合同测试之一；无测试时是否在 MR 说明**风险与手工验证步骤**。
    - Code Review：是否主动拆小 MR、回应评论是否改到点上；同一类 comment 重复出现是否沉淀为团队规范。
- **常见证据**：MR 讨论与 checklist、CI 构建与覆盖率截图、Sentry/日志关键词检索结果（脱敏）、上线单/发布群记录。

### 前端开发工程师 / 实习

- **典型产出**：组件或页面 MR；与设计稿逐屏对照说明；Story/截图或录屏；性能或无障碍相关 before/after。
- **特殊评价指标**
  - **工程化（及格线）**
    - 状态：全局 store 是否**克制**；跨组件通信是否可追踪（props/context/事件），避免隐式全局单例污染。
    - 副作用：请求取消（Abort/unmount）、竞态（后发先至）是否在列表/搜索等场景处理或注明已知局限。
    - 类型：关键 DTO 是否与后端契约对齐（生成类型或手工维护处有注释）。
  - **体验与兼容性**
    - 加载/空/错三态是否与设计一致；错误是否**可恢复**（重试、返回）而非死白屏。
    - 目标浏览器或 App WebView 是否**真机列表**验证；已知不兼容是否在 README 或 MR 写明。
    - 无障碍：关键按钮是否可键盘操作、焦点管理是否可接受（按团队标准）。
  - **与设计协作**
    - 是否能列出**实现缺口**（缺态、动效参数、极端数据量）并推动设计补充，而不是默默按错实现。
    - 走查：是否按「模块→交互→边界」顺序自检，附对照表或截图打点。
  - **性能（加分项按岗位）**
    - 长列表/大表是否虚拟化或分页；图片是否合适压缩与懒加载。
    - 包体：是否做过一次 bundle 分析或按需引入；首屏关键资源是否优先。
- **常见证据**：PR 附件截图、Lighthouse/Web Vitals 报告、设计走查文档链接、真机型号线表。

### 全栈工程师 / 实习

- **典型产出**：端到端竖切功能（前后端 MR 或单体仓库多目录）；联调环境与**数据种子**说明；简短架构/数据流一页纸。
- **特殊评价指标**
  - **竖切一致性**
    - 类型与枚举、分页字段、时间时区、错误码是否在**两端同源**（生成代码或单一 truth 表）。
    - 权限：前端隐藏与后端鉴权是否**同时**落实，而非只挡 UI。
  - **责任边界**
    - 线上问题是否能先分类：网络/CORS、网关、BFF、DB、前端态；是否把抓包/日志片段带给正确 owner。
  - **部署与链路**
    - 是否在本地或一体化环境跑通：登录态→主流程→异常路径；提测单是否写清**测试账号与数据前提**。
  - **交付颗粒度**
    - 是否拆里程碑（接口冻结→联调→UI  polish），避免 giant MR。
- **常见证据**：联调聊天记录摘要、E2E/手工测试步骤、发布前后对比截图、接口快照（Postman/示例 JSON）。

### 移动端开发（Android/iOS/RN 等）/ 实习

- **典型产出**：可安装构建或 TestFlight/内测包说明；变更列表与**权限/隐私**说明；崩溃与 ANR 处理记录。
- **特殊评价指标**
  - **机型与权限**
    - 是否维护**最低 OS**与测试矩阵；相机/定位/相册等是否在拒绝授权时有降级路径。
    - 后台任务、推送、省电策略是否在目标 ROM 上验证或标注已知限制。
  - **性能与稳定**
    - 是否有**卡顿/内存**基本分析（Profiler、Leak 工具）；启动与首屏是否有可复述优化点或「未优化原因」。
    - 崩溃：是否附堆栈、复现率、修复 MR；是否区分线上必现与实验开关问题。
  - **合规与上架**
    - 隐私弹窗、数据收集是否与产品/法务文案一致；第三方 SDK 清单是否更新。
- **常见证据**：Firebase/Crashlytics 趋势（脱敏）、真机测试表、应用商店审核备注（若可公开）。

### 软件开发实习生（未定前后端）

- **典型产出**：带 description 的小步 PR；学习笔记或「新人 FAQ」贡献；缺陷修复或工具类 MR。
- **特殊评价指标**
  - **排障独立性**：求助帖是否含环境、复现步骤、已尝试假设、相关日志/链接；是否先查过文档/issue。
  - **迭代速度**：同类任务（如 CRUD+单测、简单 bug）第二次是否**明显少轮** review。
  - **工具链**：是否在团队约定时间内独立：分支策略、rebase/merge、lint、本地构建、打点和查日志。
  - **估时可信度**：是否愿意给出区间并在阻塞时**尽早**调整预期。
- **常见证据**：PR 粒度与迭代轮次、mentor 周反馈、复盘里列出的可复用 checklist。

### 嵌入式软件实习生（侧重应用层时在软件文件中评价）

若为 **Linux 驱动/RTOS 应用**：

- **典型产出**：模块设计小注或头文件约定；接口与**资源生命周期**（fd、mutex、ISR 边界）说明；Stress/长时间跑测记录（若有）。
- **特殊评价指标**
  - **实时与安全边界**：ISR 与工作队列分工是否清晰；malloc/日志是否在禁止上下文使用。
  - **可测试性**：硬件不可用时是否有 mock/仿真路径；关键状态机是否有表驱动或断言。
  - **可追溯**：版本与配置（defconfig/feature）是否可一一对应；现场问题是否有**最小复现**步骤。
- **常见证据**：JTAG/串口 log 摘要、代码评审里对竞态的讨论、与硬件同事的对照表。
